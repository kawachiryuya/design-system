# GunMaaS リブランディング — 将来ビジョンと広域連携の設計方針

## 1. 地域MaaSの現状と課題

日本全国で地域MaaSが乱立している。令和7年度だけで29事業が国交省に採択され、北海道から沖縄まで各地域で独自のMaaSが展開されている。

ユーザー視点での問題は、行き先ごとに別のサービスを探し、別のアカウントを作り、別の使い方を覚えなければならないこと。地元住民にはいいが、県をまたぐ移動や旅行者にとっては断片化が激しい。

---

## 2. 地域MaaSを束ねる3つのレベル

| レベル | 形態 | 概要 | ユーザー体験 |
|--------|------|------|-------------|
| Lv.1 | ディレクトリ型 | どの地域にどんなMaaSがあるかの情報ポータル | 情報は得られるが、各MaaSを個別に使う |
| Lv.2 | メタ検索型 | 横断検索＋情報提示。購入は各MaaSに遷移 | 検索は1箇所、購入は分散（Skyscanner型）|
| Lv.3 | 統合プラットフォーム型 | 1アカウント・1決済で全国のチケットを購入可能 | 完全にシームレス（Amazonマーケットプレイス型）|

**Lv.2とLv.3の核心的な違いは「誰のアカウントで、誰が決済するか」。**

- Lv.2: 親サイトで検索するが、購入時に各地域MaaSに遷移。アカウント・決済は地域ごとにバラバラ
- Lv.3: 1つのアカウント・1つの決済手段で、どの地域のチケットも買える。プラットフォームが取引の主体として仲介する

---

## 3. JR東日本の戦略としての「レベル2.5」

JR東日本はLv.3に必要なアセットの大部分を既に保有している。

**既存アセット:**
- モビリティ・リンケージ・プラットフォーム（GunMaaS等の基盤）
- JRE ID（アカウント基盤）
- Suica（決済・ICカード基盤）
- JRE POINT（ポイント基盤）
- JRE GO / えきねっと（予約基盤）
- 管内の地域MaaS事業者との既存連携

**推奨戦略:**
- JR東日本管内 → Lv.3（JRE ID + Suica + 地域MaaSチケットのワンストップ購入）
- JR他社エリア → Lv.2（横断検索＋情報提示。購入は各社サービスへ遷移）
- 将来的にJR各社間連携の深化に応じてLv.3に段階的引き上げ

## 4. JRE GOと地域MaaSの統合ロードマップ

### Phase 1（すぐできる）: 導線だけ作る

JRE GOの新幹線予約完了画面に、到着駅に応じた地域MaaS情報を動的に表示。タップすると各MaaSサイト（g3m.jp等）に遷移。

- システム間連携なし、UIにリンクを追加するだけ
- 「駅→地域MaaSのマッピングテーブル」が必要（技術的には軽微）
- 購入体験は分断されるが、導線としての価値はある

### Phase 2（中期）: チケット販売APIの整備

モビリティ・リンケージ・プラットフォームにAPIを整備し、JRE GO内で地域MaaSチケットを購入可能にする。

**必要なAPI:**

| API | 機能 | 説明 |
|-----|------|------|
| チケットカタログAPI | 何が買えるか | エリアIDでリクエスト → チケット一覧を返す |
| チケット購入API | 買う | JRE IDとチケットIDで購入リクエスト → チケット発行 |
| チケット参照API | 確認・利用する | JRE IDで保有チケット一覧を返す（直接購入＋JRE GO経由の両方）|

**ポイント:**
- 決済はJRE GO側で処理（既存のクレカ/Suica）。プラットフォーム側はチケット発行のみ
- APIを標準化すれば、新しい地域MaaSが追加されるたびにJRE GOから自動アクセス可能
- 各地域MaaSのWebサイトは引き続き単独で動作（地元住民の利用に影響なし）

**事業者連携の課題:**
- 販売手数料の設計（JRE GOの手数料をどう設定するか）
- チケット情報の標準フォーマット策定
- 予約制チケットのリアルタイム在庫同期（フリーパス等は不要）

### Phase 3（長期）: フル統合

- Suica連携（地域バスでのICタッチ乗車）
- マイナンバー情報の横断利用（地域割引の自動適用）
- 統合チケットホルダー（新幹線＋地域チケットを1画面で）
- GTFS-Flex対応によるデマンド交通の経路検索統合

---

## 5. 統合後のGunMaaSの役割再定義

JRE GOに経路検索が統合されると、GunMaaSの役割が変わる。

### JRE GOに移る機能
- 経路検索（検索タブの機能）
- チケット提案・購入（購入フロー）
- チケットホルダー（チケットタブの機能）

### GunMaaSに残る機能
- 行き先の発見（ホームタブのマップ＋行き先オブジェクト）
- 地域コンテンツ（行き先詳細ページ: 基本情報・チケット・周辺スポット）
- デマンド交通の発見・予約（マップのゾーン表示＋予約フロー）
- JRE GOへの経路検索の入口（検索バーからJRE GOに遷移）

**GunMaaSは「群馬の移動ツール」から「群馬の行き先を発見し、体験を計画するためのプラットフォーム」に進化する。**

「移動をつなぐ」インフラはJRE GOが担い、「行きたいをかなえる」コンテンツはGunMaaSが担う。

## 6. デマンド交通の扱い

### 現状の課題
GTFS-Flex未対応のため、デマンド交通は経路検索に乗らない。JRE GOに経路検索を移植してもデマンド交通はルート候補に出てこない。

### 当面の方針（GTFS-Flex未対応期間）
- GunMaaSがデマンド交通の唯一の入口。マップのゾーン表示→タップ→予約
- JRE GO側では、交通空白地帯の経路検索結果にコンテキストカードを表示（「この地域ではGunMaaSの呼べば来るバスが利用できます → GunMaaSで予約」）

### 将来の方針（GTFS-Flex対応後）
- 経路検索結果に「るんるんバス 14:30発（要予約）」が候補として出る
- 予約フローをJRE GO内で完結させるかGunMaaSに遷移させるかはAPI整備状況次第

### 変わらない価値
デマンド交通には「エリア発見」機能が必要。「自分の家の近くにデマンド交通があること自体を知らない」ユーザーに対して、マップ上のゾーン表示で視覚的に発見させる。これは経路検索UIでは実現できず、GunMaaSの独自価値として残り続ける。

## 7. マップの県境制御

### 原則
**「行き先オブジェクトの管理責任は所在地の県が持つ」**

GunMaaSは群馬県内の行き先だけを登録・管理する。県外スポット（軽井沢等）は登録しない。

### 理由
- 隣県MaaS（長野版MaaS等）が立ち上がった場合に、行き先情報の二重管理が発生する
- 「群馬からのアクセスが多い」という理由で県外スポットを登録すると、管理責任の境界が曖昧になる
- 県境で明確に切ることで、隣県MaaSとの連携が将来的に破綻しない

### マップの表示制御
- **ピン:** 群馬県内の行き先オブジェクトのみ表示
- **路線:** 群馬県内の区間のみ表示。県境でフェードアウト
- **デマンドゾーン:** 市町村単位のため県境問題なし
- **県外エリア:** グレーアウトまたは薄い表示。タップすると「このエリアは〇〇MaaSの管轄です → 〇〇MaaSを開く」を案内

### 県外への移動ニーズの解決
群馬から軽井沢に行きたい人は、GunMaaSの検索バーに「軽井沢」と入力 → GunMaaSには行き先オブジェクトがないため、JRE GOの経路検索に遷移 → 経路を表示。軽井沢の行き先情報は将来的にNaganoMaaSが提供。

**GunMaaSのマップは「群馬の行き先ポータル」に徹し、県外移動はJRE GOに委ねる。**

## 8. GunMaaSリブランディングとの接続

今回のリブランディングで設計した以下の要素は、将来ビジョンと整合している：

| 設計要素 | 将来ビジョンとの関係 |
|----------|---------------------|
| OOUI（行き先オブジェクト中心） | JRE GOに経路検索が移っても、地域コンテンツとして価値を持ち続ける |
| チケット提案セクション | Phase 2でJRE GOに組み込む際のUIモデルとしてそのまま参照可能 |
| プログレッシブ・オンボーディング（JRE ID/Suica/マイナンバー） | 全国の地域MaaSに共通する仕組みとして標準化可能 |
| コンテキストカード（デマンド交通補完） | JRE GO側にも同じ概念で組み込める |
| マップのレイヤー構造（行き先/路線/デマンドゾーン） | 県境制御と隣県連携の基盤として機能 |
| デザインシステム | 他の地域MaaSにも展開可能な共通UIパターンとして提供できる |

**提案書では「単なるUIリニューアルではなく、JRE GOとの統合と地域MaaS間連携を見据えたアーキテクチャ設計」として位置づけることで、提案の説得力が大幅に増す。**
